There is one remaining realtime/cache navigation issue.

Everything else is working correctly, so please do not make broad architectural changes.

Reproduction:

1. Open a community chat.
2. Navigate from Communities to the Home page.
3. While on Home, another user sends a new message to that community.
4. Navigate back to Communities.
5. On this FIRST return, the new message is NOT visible.
6. Navigate to Home again.
7. Navigate back to Communities a SECOND time.
8. Now the missing message appears instantly.

This strongly suggests that on the first return, the incremental catch-up fetch successfully gets the new message and updates the module/global message cache, but the currently mounted CommunityChat React `messages` state is not being updated/rendered.

Then on the second navigation, CommunityChat initializes from the already-updated cache, so the message appears.

Please investigate this exact flow.

Inspect:
- CommunityChat.tsx
- lib/communities/cache.ts
- CommunitiesPanel.tsx
- sidebarStore/page caching logic
- any layout/page persistence or navigation caching
- incremental catch-up fetch logic added in the previous fix

Check specifically:

1. When CommunityChat mounts after returning from Home with cached messages:
   - Cached messages should render immediately.
   - Incremental catch-up should fetch messages newer than the latest cached real message.
   - When that fetch completes, it MUST update BOTH:
     a. msgCache
     b. the currently mounted component's `messages` React state

2. Verify that a stale closure, mounted flag, request ID, AbortController, communityId guard, or effect cleanup is not preventing `setMessages()` on the first return.

3. Check whether the page/sidebar caching implementation keeps CommunityChat mounted but hidden instead of fully unmounting it.
   If it remains mounted, make sure returning to the Communities page triggers an incremental catch-up and updates visible state.

4. Check whether the first-return incremental fetch only calls something equivalent to:
   `msgCache.set(...)`
   without synchronizing the result into:
   `setMessages(...)`

5. Check for a race between:
   - reading cached messages
   - setting initial messages
   - incremental catch-up
   - Supabase subscription setup
   - page navigation/mount state

A late cache initialization must not overwrite newer messages fetched by the incremental catch-up.

6. Make sure the catch-up merge is atomic:

   previous/current messages
   + newly fetched messages
   → deduplicate by ID
   → chronological sort
   → update msgCache
   → update setMessages with the SAME merged result

7. Make sure the currently active communityId is checked before calling setMessages, so a request from another community cannot update the wrong chat.

Expected behavior:

Community A open
→ navigate Home
→ new message arrives in Community A
→ navigate back to Communities
→ cached Community A appears instantly
→ incremental catch-up runs immediately
→ newly missed message appears immediately on this FIRST return

The user should NOT need to navigate away and return a second time.

Important:
Do not remove the existing caching optimization.
Do not add polling.
Do not perform unnecessary full message refetches.
Do not break Supabase Realtime.
Do not make unrelated changes.

First identify and explain the exact root cause before modifying code. Then implement the smallest surgical fix and report which files were changed.
