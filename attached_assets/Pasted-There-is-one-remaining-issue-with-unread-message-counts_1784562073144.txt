There is one remaining issue with unread message counts.

Everything else is working now. Please do not make broad changes to the realtime/message caching architecture.

PROBLEM:

Unread counts are stale when messages arrive while the user is outside the Communities page.

Exact reproduction:

1. Community A currently has 3 unread messages.
2. User is on the Home page, not Communities.
3. Another user sends 5 new messages to Community A.
4. Actual unread count should now be 8.
5. User navigates to Communities.
6. The sidebar still shows 3 unread instead of 8.
7. While the user stays on Communities, another realtime message arrives.
8. The count changes from 3 → 4.

But the correct count should have been:

3
+ 5 messages received while away
= 8

then one new realtime message:

8 → 9

This indicates that the background/incremental message catch-up correctly synchronizes missing messages, but the unread-count state/cache is NOT being synchronized.

The realtime handler is then probably incrementing a stale unread count:

3 → 4

instead of starting from the authoritative synchronized count:

8 → 9.


PLEASE INVESTIGATE FIRST.

Inspect all relevant code for:

- unread count calculation
- unread count cache/store
- sidebarStore
- CommunitiesPanel
- CommunityChat
- message catch-up/incremental fetch
- Supabase Realtime message handler
- last-read/read-state logic
- APIs that return unread counts
- page navigation/cache restoration

Do not assume the fix before tracing the full unread-count data flow.


EXPECTED BEHAVIOR:

Scenario:

Community A has 3 unread.

User goes to Home.

5 messages arrive while Communities is not mounted.

User returns to Communities.

Immediately/background sync should reconcile the unread count:

3 → 8

Then if another realtime message arrives:

8 → 9


IMPORTANT:

The incremental message catch-up and unread-count synchronization need to work together.

When returning to the Communities page, do NOT rely only on the cached sidebar unread count.

The app must reconcile unread counts with the server/database.


Please determine which architecture is currently used:

A) Server stores/read-tracks an authoritative unread count.

OR

B) Unread count is calculated from:
   messages newer than user's last_read_at / last_read_message_id.

Use the existing architecture. Do not introduce a second competing unread-count system.


DESIRED FLOW:

User on Communities:
Realtime message
→ message state/cache updated
→ unread count updated appropriately

User on Home:
Messages can arrive while Communities realtime subscription is not active.

Return to Communities:
→ show cached sidebar immediately for fast navigation
→ background sync messages
→ background sync/recalculate authoritative unread counts
→ update sidebarStore/state/cache
→ UI changes from stale cached count to correct count

After sync:
New realtime message
→ increment from the NEW synchronized unread count


IMPORTANT EDGE CASE:

Do not blindly increment unread count for every message fetched during incremental catch-up if the server unread count already includes those messages.

That could double-count.

Prefer fetching/recalculating the authoritative unread count after being away, or calculate deterministically from existing last-read state.

For example, avoid:

cached unread = 3
catch-up finds 5 messages
3 + 5 = 8
server sync also returns 8
then accidentally add again

The final count must have one source of truth.


ALSO CHECK:

1. Does sidebarStore contain cached unread counts that persist across page navigation?

2. When Communities mounts again, is sidebarStore.data considered "fresh" and therefore skipping the API request because of a stale-time/cache gate?

3. If yes, cached sidebar data should still render instantly, but an appropriate lightweight background revalidation should happen when returning to Communities.

4. Does the incremental message fetch update msgCache but not sidebarStore unread counts?

5. Does the Supabase realtime handler use something like:

unreadCount + 1

against a stale closure?

If so, use a functional/current-state update after the authoritative count has been synchronized.

6. Make sure background unread synchronization updates BOTH:
- the rendered sidebar state
- the relevant sidebar cache/store

Otherwise the same bug may reappear after navigation.

7. Avoid race conditions:
If an authoritative unread-count fetch starts and then a new realtime message arrives before the fetch completes, the older fetch result must NOT overwrite the newer realtime count.

Example:

Server fetch starts → server count = 8
Realtime message arrives → UI becomes 9
Old fetch response returns → must NOT incorrectly reset UI to 8

Handle this using the existing store/revalidation architecture or merge based on message/read state rather than blindly replacing newer state.
