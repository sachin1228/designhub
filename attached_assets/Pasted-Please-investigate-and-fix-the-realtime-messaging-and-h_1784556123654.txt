PROBLEM 1 — Messages are missed when switching communities

Currently Supabase Realtime appears to subscribe only to the active `communityId`.

Example:
1. User opens Community A.
2. User switches to Community B.
3. Someone sends a message in Community A.
4. User switches back to Community A.
5. The new message does not immediately appear.
6. It may appear much later.

The likely cause is:
- Community A's realtime subscription is removed when switching to Community B.
- Messages received while A is inactive are missed.
- When returning to A, cached messages are considered fresh because `MSG_STALE_MS` is 3 minutes.
- Therefore the app does not fetch the messages that were missed.

FIX:

When `communityId` changes:

1. If cached messages exist:
   - Show cached messages immediately.
   - Do NOT show a loading skeleton.
   - Immediately run an incremental/background catch-up fetch.
   - Fetch messages newer than the latest cached REAL message using `created_at`.
   - Do this regardless of `MSG_STALE_MS`.

2. If no cached messages exist:
   - Perform the normal full message fetch.

3. Merge fetched messages with cached/current messages:
   - Deduplicate by message ID.
   - Preserve chronological ordering.
   - Update React `messages` state.
   - Update `msgCache`.
   - Update `msgFetchedAt`.

4. Keep Supabase Realtime subscribed to the CURRENT active community so messages appear instantly while the user is viewing that chat.

Do NOT add aggressive polling.

Desired flow:

Active community
→ Supabase Realtime delivers messages instantly

Switch to another community
→ show its cached messages instantly
→ immediately incremental-fetch missed messages
→ subscribe to its realtime channel

Switch back to previous community
→ show cached messages instantly
→ immediately incremental-fetch messages received while inactive
→ subscribe to realtime
