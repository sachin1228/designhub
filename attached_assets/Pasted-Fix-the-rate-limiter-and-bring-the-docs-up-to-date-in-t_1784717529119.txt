## 1. Fix the rate limiter (apps/web/lib/auth/rate-limit.ts)

The current rate limiter uses an in-memory Map, which does not work correctly 
in serverless/multi-instance deployments (e.g. Vercel) — each instance has its 
own memory, so attempts get spread across instances and the limit never 
triggers reliably.

Replace it with a Redis-backed rate limiter using Upstash:
- Install @upstash/ratelimit and @upstash/redis
- Use Ratelimit.slidingWindow() to preserve the existing limits already used 
  in apps/web/app/api/auth/login/route.ts (10 requests / 15 min per IP, 
  20 requests / 15 min per email)
- Keep the same function signature/usage pattern where possible so callers 
  don't need to change, or update all call sites if the interface changes
- Add UPSTASH_REDIS_REST_URL and UPSTASH_REDIS_REST_TOKEN to 
  apps/web/.env.example with comments on where to get them (Upstash dashboard)
- Fail safely if Redis is unreachable — decide and document whether it should 
  fail open (allow the request) or fail closed (block it), don't leave it 
  ambiguous
- Apply the same fix anywhere else in the codebase that reuses this rate 
  limiter, not just the login route
